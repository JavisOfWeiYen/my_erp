A · 改版理由
背景 為什麼有 v2

v1 的盲點：列了一堆「可以做的分析與模型」，但沒回答最重要的問題——demo 那 5 分鐘要讓觀眾看到什麼？

v1 的四層結構（描述 → 毛利 → 預測 → Agent）邏輯清楚，但容易讓實作落入「把每層的清單都做完」的陷阱。對 AI agent demo 而言，做了 8 個模型但講不出故事，比做了 3 個但每個都有戲劇張力差。

v2 改成劇本驅動：先寫好 5 個 demo 劇本，每個劇本盤點需要什麼資料、什麼 endpoint、什麼模型；任何不在劇本裡的東西一律砍掉或延後。

核心思維 Demo 的單位是「對話」不是「功能」。觀眾記得住的是「AI 看出 5 月毛利下滑是因為原料漲價」，不是「我們上線了 8 個 ML endpoint」。
原則 v2 設計原則
  1. 劇本先於功能。每個能力都要對應到至少一個 demo 劇本中的 turn，否則不做。
  2. 可解釋 > 精準度。Logistic Regression 講得出「為什麼這個客戶會流失」比 XGBoost 的 F1=0.87 更有 demo 張力。
  3. 資料是劇本的一部分。Seed 不是亂塞，是編劇——某客戶在某月開始流失、某商品 Q4 暴漲、某供應商曾漲價 15%。
  4. Agent 可以說「我不知道」。Demo 要呈現 agent 邊界，比假裝萬能更可信。
  5. 寫操作要 human-in-the-loop。建單、確認、作廢一律 agent 出草稿、user 拍板。

TOOLget_monthly_report(year=2026, month=5) → 營收 NT$1.24M（vs 4 月 1.18M, +5%）
TOOLget_gross_margin(year=2026, month=5) → 毛利率 18.2%（vs 4 月 21.5%）
TOOLget_cost_trend(months=6) → SUP-003 cost_price 4 月 NT$45 → 5 月 NT$52（+15.6%）

a) 售價調整：這 3 項建議跟漲 6-8%（仍低於同業）。

Demo 賣點 這是整個 demo 最有戲劇張力的場景——使用者只問一個「為什麼」，agent 自己拆解出 4 個工具呼叫、組合成完整因果鏈與行動建議。
C · 前置：資料故事
資料 資料故事設計（Seed Blueprint）

五個劇本所有「巧合」都來自 seed data 的事先編劇。Seed generator 不是 faker，是劇作家。

核心結構

  • 時序跨度：18 個月（2024-12 ~ 2026-05），讓 12 月季節性訊號清楚。
  • 規模：30 商品 / 6 分類、25 客戶、12 供應商、8 員工（含 5 業務）、約 800 銷貨單、400 進貨單、120 盤點調整、200 AR、120 AP、180 收付款記錄。
  • 密度：每月平均 ~45 銷貨、~22 進貨；冬季旺月（11-2 月）+30%，夏季淡月（6-8 月）-15%。

編劇 — 供應商故事

  • 大宇 (SUP-003)：2026-04、2026-05 連續漲價（5%、10%）——觸發 S5 故事。lead_time 7 天。
  • 佳能 (SUP-005)：穩定、價格從未調整、lead_time 10 天，但我們平均只用 22 天才付款（合約 45 天）——S4 延付空間。
  • 安捷 (SUP-008)：跟大宇有重疊品項但價格略高 3%——S5 替代供應商建議。

實作方式

  • 新 script：app/scripts/seed_demo_data.py（idempotent、可 reset）。
  • 用 faker 補背景客戶/商品；劇本角色寫死。
  • 分三段：（a）static fixtures（角色、商品、供應商）；（b）時序生成（每月 loop，依角色行為 sample）；（c）after-the-fact 注入（在 5 月份對 SUP-003 cost 動手腳）。
  • 每次跑 --reset 會清掉前一次 demo 資料、保留真實使用者資料（用 is_demo 旗標或專屬 user 隔離）。
注意 Demo seed 與真實使用者操作不能混在同一 DB schema 上；建議用 --demo profile（不同 SQLite 檔）或加 is_demo 欄位過濾。

E · ML 模型
L3 ML 模型選擇（精簡版） 統一原則：可解釋 > 精準度

v2 大砍 v1 候選清單——只留下劇本用得到、且解釋性強的。

能力v2 選擇砍掉的候選理由
需求預測（商品級） Holt-Winters（statsmodels） Prophet、LightGBM Prophet stan 編譯肥；LightGBM 對 demo 而言精準提升不值得複雜度
需求預測（公司級） Holt-Winters — 同上、且資料聚合後雜訊小、HW 已足
客戶分群 RFM 分位切割（純規則） K-means、DBSCAN RFM 555 比 cluster #3 好懂 10 倍；demo 講不出 K-means 的價值
流失預測 Logistic Regression（sklearn） Random Forest、XGBoost 係數能直接讀「哪個特徵推高流失機率」
AR 逾期風險 Logistic Regression XGBoost 同上
庫存異常 z-score（純統計） Isolation Forest z-score 講「比平均高 3σ」比 IF 講「異常分數 0.72」直觀

模型生命週期（簡化）

  • 訓練：app startup hook + 每日 cron（用 APScheduler 或外部 cron）。模型 pickle 到 backend/app/ml/models/。
  • 推論：endpoint 直接 load 並 cache。
  • 沒有 MLflow、沒有 feature store——demo 不需要。
  • 新 endpoint：
    • GET /forecasts/demand?product_id=&horizon=3
    • GET /forecasts/revenue?horizon=3
    • GET /analytics/customer-rfm
    • GET /analytics/churn-risk?top=20
    • POST /analytics/ar-risk（body 含一張未確認的 SO 資訊）
    • GET /analytics/shrinkage-anomaly?lookback_months=6
    • GET /analytics/reorder-suggestions（產出補貨清單，S3 直接吃）
解釋性原則 所有 ML endpoint 都要回傳說明欄位，不能只給數字。例如 churn-risk 回傳： 
{ "customer_id": 12, "risk": 0.78,
  "reasons": [
    "已 45 天沒下單（平均 18 天）",
    "金額同比下滑 62%",
    "主購品項 LED-A100 我們近期漲價"
  ] }
這讓 agent 不只說「祥豐流失風險 78%」、而是「祥豐流失風險高，因為……」——這才是 demo 張力。
F · Agent 整合
L4 AI Agent 架構與 Tool Schema Anthropic Claude tool use

架構

  • 新後端模組 app/agent/，內含 tools.py（工具註冊）、prompts.py（system prompts）、runner.py（multi-turn loop）。
  • 新 endpoint POST /agent/chat：body 帶 messages、user_id、可選 mode（chat / briefing / analyze）。回傳 streaming 或 batch。
  • 用 Anthropic SDK（Claude Sonnet 4.6 即可、無需 Opus）。啟用 prompt caching 把 system prompt + tool schema 快取。
  • 權限：Agent 用呼叫者的 JWT 操作；寫入工具額外做 RBAC（建草稿要 admin/manager/warehouse；agent 不能直接 receive/confirm）。

Tool Schema 範本

{
  "name": "get_sales_trend",
  "description": "取得銷售趨勢，可按月/類別/客戶/業務員分組，並可選同比比較。\n用於：回答『銷售變化』、『哪段時間衰退』、『誰賣得最多』等問題。",
  "input_schema": {
    "type": "object",
    "properties": {
      "start_date": { "type": "string", "format": "date" },
      "end_date": { "type": "string", "format": "date" },
      "group_by": { "enum": ["month", "category", "customer", "salesperson"] },
      "compare": { "enum": ["none", "mom", "yoy"] }
    },
    "required": ["start_date", "end_date"]
  }
}

系統提示要點

  1. 角色定義：「你是 my_erp 內建分析助理，協助使用者解讀進銷存與會計資料、提出洞察、必要時建立草稿單據。」
  2. 邊界：列出可讀/可寫範圍。寫入動作只能建 draft，禁止直接 receive/confirm/void。
  3. 回答風格：先給結論、再給數字、再給建議。避免讀稿式逐條工具呼叫；回應要像同事不像 SQL。
  4. 不確定時要明說：「資料只有 3 個月，預測信心較低」比假裝確定好。
  5. 使用者語言：預設繁中。
