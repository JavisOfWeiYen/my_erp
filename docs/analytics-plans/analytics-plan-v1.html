A · 起點
原則 設計原則

為什麼要這樣分層？

  1. AI agent demo 的價值 = 它能回答的問題 × 它能做的決策。所以重點不是堆 ML 模型，而是讓 agent 有「資料 + 工具」可用。
  2. 由易到難、由描述到預測。先把現有資料能講的故事講完，再進 ML。
  3. 每一層都要能單獨 demo，不必所有都做完才能展示。
關鍵 insight Layer 1 + 2 已經能撐起非常驚艷的 demo——AI 不靠 ML 也能做「描述、診斷、跨表推理」。Layer 3 才需要 ML，Layer 4 才需要 LLM 整合。
B · 四層分析
L1 描述性分析（What happened） 純 SQL／pandas · 無 ML

純聚合，不需要任何 ML，靠的就是現有資料。建議當基礎打好，AI agent 可以直接讀。

分析用什麼資料Demo 問句
銷售趨勢分解 sales_orders（confirmed）按月/類別/客戶/業務 「Q2 銷售比 Q1 多多少？哪一類成長最多？」
ABC / Pareto 分析 sales_order_items 累計營收排序 「貢獻 80% 營收的客戶有幾個？」
庫存周轉率 (avg 月銷量 / 期末庫存) 「哪些品項超過 90 天沒動？」
應收帳款表現（DSO） (AR balance / 月銷貨) × 30 「平均收款天數？哪個客戶最會拖款？」
應付帳款表現（DPO） AP balance / 月進貨 「我們對哪個供應商付款最快？」
庫存損耗分析 stock_adjustments 按 reason 聚合 「報廢最多的是哪幾項？金額多少？」
業務員業績排行 已有 /salesperson-report，加進步/退步比較 「業務 A 的銷售環比是進步還是退步？」
現金流現況 AR 帳齡 + AP 排程預期收支 「下月預期淨現金流是多少？」
前置條件 豐富 seed data。沒有 12 個月跨度的資料，這些分析都看不到趨勢。
L2 毛利分析（What's profitable） 小幅 schema 改動 · 無 ML

獨立列一層，因為要先做 schema 改動：sales_order_items.unit_cost（confirm 當下快照 product.cost_price）。

報表內容
商品毛利率排行哪個商品高毛利但低銷量？哪個薄利多銷？
客戶毛利率排行哪些客戶看似大買家但毛利低（議價兇）？
業務員毛利貢獻不只看營收，看實際毛利
毛利率時序是否在惡化？是進貨成本上升還是售價下降？
為什麼重要 這層做完，AI agent 才能回答「真正賺錢的客戶是誰」，不只是「買最多的客戶是誰」。

成本配對方法選擇

  • 方案 A · 快照成本（建議）：銷貨 confirm 當下，快照 product.cost_price 寫入 sales_order_items.unit_cost。簡單、穩定。
  • 方案 B · FIFO：每次銷貨從最早的進貨批次扣，需要 batch-level 庫存表 stock_lots。大改動。
L3 預測模型（What will happen） scikit-learn / statsmodels

進入 ML。挑「簡單模型、解釋性強、demo 效果明顯」的組合。

3.1 需求預測（Demand Forecasting） 中

  • 目標：每個商品未來 1–3 個月銷量預測。
  • 模型選擇（由簡到繁）：
    1. Naive seasonal：去年同月銷量 + 趨勢項。推薦先做這個 demo——零依賴、可解釋、12 個月 seed data 就夠。
    2. 指數平滑（Holt-Winters）：statsmodels 一行搞定，含季節性。
    3. Prophet：好處是處理假日、缺值；缺點是 stan 編譯肥。
    4. LightGBM with lags：要做特徵工程，但能納入「同類商品」、「促銷期」等外生變數。
  • 新 endpoint：GET /forecasts/demand?product_id=&horizon=3
AGENT下個月你的 A 商品預估賣 250 個，目前庫存 80，建議補貨。

3.2 補貨建議（Reorder Suggestion） 易

  • 公式：reorder_point = demand_forecast(lead_time) + safety_stock
  • 資料：3.1 的預測 + 供應商 lead time（需新增 suppliers.lead_time_days 或 config 預設值）
  • 不算 ML，是 3.1 的應用，但 demo 效果直觀。

3.3 客戶 RFM 分群 + 流失風險 中

  • RFM：Recency（最後購買距今）、Frequency（過去 N 月下單次數）、Monetary（過去 N 月營收）。三維度各切分位，得到 R/F/M 三碼。555 = VIP。
  • 流失定義：N 天沒下單 = churned。可訓 Logistic Regression / Random Forest。
  • 新 endpoint：GET /analytics/customer-rfm、GET /analytics/churn-risk
AGENT本週要打電話關心的客戶：[祥豐]、[泰昌]、[宏億]——這三家上月還在前 10，這月完全沒下單。

3.4 應收帳款逾期風險預測 中

  • 特徵：客戶歷史平均逾期天數、歷史逾期率、capital、訂單金額、payment_terms、季節月份。
  • 模型：Logistic Regression（最可解釋）或 XGBoost。
  • 訓練資料：歷史 AR (status=paid)，配對 payment 時間，計算實際付款日 vs due_date 是否逾期。
  • 新 endpoint：POST /analytics/ar-risk
AGENT這筆銷貨單建議要求預付——客戶歷史逾期率 60%，且本筆金額較大。

3.5 庫存異常偵測（盤點數據） 易

  • 方法：Isolation Forest 或統計法（過去 6 個月 adjustment 的 z-score）。
  • 新 endpoint：GET /analytics/shrinkage-anomaly
AGENT商品 X 報廢率比同類高 3 倍，可能要查。

3.6 銷售總額預測（公司層級） 易

  • 目標：下個月／下季營收預測。比 3.1 簡單，因為是聚合時序、雜訊小。
  • 模型：Holt-Winters。
  • 新 endpoint：GET /forecasts/revenue?horizon=3
建議順序 3.6 → 3.1 → 3.2 → 3.5 → 3.3 → 3.4。先做最簡單、最聚合的，再做需要客戶級訓練資料的。
L4 AI Agent 整合（真正的 demo 賣點） LLM tool use · Anthropic / OpenAI SDK

前三層做的是「準備好工具」。這層是「讓 AI 用工具」。

4.1 工具化 API

把上述所有 endpoint 包成 LLM tool use 格式（OpenAI Function Calling / Anthropic tool use schema），讓 agent 知道：

  • 有哪些查詢可用、需要什麼參數
  • 工具回傳格式
  • 何時該用哪個

4.2 對話式分析

AI agent 接收自然語言問題，自行決定要呼叫哪些 endpoint、組合資料、回答。

USER為什麼 5 月利潤下滑？
AGENT [1] 呼叫月報 → 確認營收持平
[2] 呼叫毛利報表 → 發現毛利率掉
[3] 比較進貨成本 → 看出原料漲價
結論：營收沒掉，是進貨成本上升 8% 吃掉毛利。

4.3 主動式洞察（Daily Briefing）

每天早上自動產出 3–5 句摘要：

AGENT早安。昨日新增 3 筆銷貨單共 NT$45,200。注意：客戶「祥豐」帳款已逾期 32 天且歷史逾期率 60%，建議今日致電。商品「LED-A100」庫存只剩 12 件，依當前銷售速度將於 5 月 28 日缺貨。

4.4 多步驟代辦（Agentic Actions）

AI agent 不只回答，還能執行：

AGENT建議補貨 A 商品 200 件。要我幫你建草稿進貨單嗎？
USER好。
AGENT已建立進貨單 PO-20260520-0003（draft）。等你確認後執行 receive 即會入庫。
注意 Layer 4 一定要設好 權限邊界：agent 能讀什麼、能寫什麼、什麼動作要 user 二次確認。對 demo 而言，建議寫入操作（建單、確認、作廢）都要 human-in-the-loop。
C · 落地計畫
路線 建議實作順序（Stages）

每一階段做完都是一個可獨立展示的里程碑。建議照 S1 → S2 → S3 → S4 走。

STAGE 1
豐富 seed data
~ 1 session
跨 12 月、含各種情境（逾期、作廢、調整）。馬上能展示現有報表的真實感。
STAGE 2
毛利 + L1 補完
~ 1–2 sessions
毛利分析（schema + endpoint）+ L1 缺的描述性 endpoint（DSO、ABC、周轉率）。
STAGE 3
預測模型
~ 2–3 sessions
從 3.6 → 3.1 → 3.4 → 3.3 依序。每加一個就 demo 一次。
STAGE 4
AI agent 整合
~ 2–3 sessions
工具化、對話 demo、Daily briefing、代辦動作。真正的 demo 賣點。
關鍵節點 S1 是 S2/S3 demo 效果的乘數——資料豐富了，後面的分析才有故事可講。

資料 資料前置條件

每個 Layer 對 seed data 的要求。

Layer對資料的要求Seed 需要哪些情境
L1 跨 6–12 個月的銷貨、進貨、AR、AP、調整紀錄 每月至少 10+ 筆銷貨、5+ 筆進貨；含逾期 AR、已作廢付款、各原因的庫存調整
L2 歷史進貨價格有變化（讓毛利率有故事） 同一商品的進貨單 cost_price 在 12 個月內有 2–3 次調整
L3 季節性／趨勢／客戶分群的訊號要在資料裡 商品銷量帶季節性；客戶可分為「常買」「偶買」「流失」三群；AR 逾期率因客戶而異
L4 L1–L3 已就緒，agent 才有工具可呼叫 無額外資料要求，但要把 endpoint 包成 tool use schema
關鍵 不要 random 亂塞資料。Seed 要設計成「有故事可講」的——例如某客戶在某月開始流失、某商品在 Q4 賣得特別好、某供應商曾漲過價。AI agent 找到這些 pattern 才有 demo 張力。
