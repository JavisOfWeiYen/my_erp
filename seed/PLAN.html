A · 起點
原則 目標與設計原則

產出讓 AI agent demo 講得出戲劇張力的 ERP 資料——不是亂塞 faker、而是編劇。

5 條設計原則

  1. API 優先：透過 backend 提供的 endpoint 寫入，業務邏輯（draft → confirm、receive 自動加庫存與寫 cost_price、confirm 自動建 AR + 寫 unit_cost 快照）全部走過一遍，跟真實使用者操作一致。
  2. SQL 只用於 timestamp backdate：要做 18 個月時序，必須把 ordered_at / confirmed_at / received_at 推到歷史日期。API 沒提供此能力 → 用 raw SQL UPDATE 直接改時間欄位（限 seed 用途）。
  3. 故事大於量：30 客戶比 300 客戶更有 demo 張力；劇本角色（祥豐流失、和欣議價兇、大同 VIP）必須在資料裡能讀出來。
  4. 真實價格範圍：GPU 的進價、售價、漲跌都對應 2024-2026 真實市場狀況（NVIDIA RTX 50 系列上市、AI 熱潮、4090 清庫存）。
  5. 可重跑、可中斷：--reset 全砍重來、--resume 從中斷處繼續、--dry-run 只列計劃。
為什麼 API 優先 若直接 raw SQL 塞入，會跳過 confirm() 的核心邏輯（扣庫存 / 寫 unit_cost / 建 AR），這些是 demo 後續分析 (毛利、AR 帳齡) 賴以為生的副作用。API 雖然慢，但每一筆都是有效的業務流，後續報表自動正確。
資料隔離 Seed data 跟你手動建的真實測試資料若混在同一 DB，--reset 會一起清掉。建議用獨立的 seed.db（透過 DATABASE_URL env var 切換），或先 git 備份 dev DB 再跑。
B · 結構
檔案 資料夾結構

在 my_erp/seed/ 下，跟 backend/、frontend/ 平行，是獨立工具。

seed/ ├── PLAN.html # 本計劃書 ├── README.md # 快速使用、CLI 範例 ├── requirements.txt # httpx, python-dotenv（少量依賴） ├── seed.py # CLI 主入口 (--reset / --resume / --dry-run) │ ├── config/ # 純資料：劇本、靜態清單 │ ├── __init__.py │ ├── products.py # ~40 GPU SKU、進價、售價、上市/退市時間 │ ├── people.py # 50 員工（姓名、角色、入職、業績檔次） │ ├── customers.py # 30 客戶（含 VIP / 流失 / 議價兇 / 逾期戶劇本） │ ├── suppliers.py # 8 供應商（含漲價 / 缺貨情境） │ └── stories.py # 時序事件編劇表 │ ├── generators/ # 邏輯：把資料推進 backend │ ├── __init__.py │ ├── api_client.py # httpx wrapper、auth、重試 │ ├── seed_setup.py # 種角色、初始 admin（idempotent） │ ├── seed_people.py # 透過 /users 建 50 人 │ ├── seed_catalog.py # 分類 / 商品 / 供應商 / 客戶 │ ├── seed_timeline.py # 主迴圈：逐月生成進銷貨 │ ├── seed_payments.py # 收付款入帳 + 部分作廢 │ ├── seed_adjustments.py # 盤點調整 │ └── timestamp_backfill.py # raw SQL 把 timestamps 推到歷史 │ └── data/ └── gpu_prices.json # 真實 GPU 價格時序（網路查得）
為什麼分 config / generators config 是「劇本與資料」、generators 是「執行邏輯」。日後要調整某客戶故事或某商品價格，只改 config；要改 API 用法或時序邏輯，只改 generators。
C · 公司
公司 公司設定

業態：NVIDIA / AMD GPU 通路 + 系統整合商

  • 主營：消費級遊戲卡（NVIDIA RTX 50/40 系列、AMD Radeon）+ 工作站（RTX Ada 系列）+ 資料中心 AI 卡（H100 / A100 / L40S）。
  • 客戶結構：B2B 為主（電競通路、系統整合商、AI 公司、雲服務商、學術機構），少量 B2C（散戶）。

  • 定位：中型代理商，毛利率 15-30%（資料中心高、消費級低）。

姓名生成

用台灣常見姓氏 + 名字混搭（張、王、李、陳、林、黃、吳、劉、蔡、楊、許、鄭、謝、洪、郭、邱、曾、廖、賴、徐 ……）。劇本角色給固定姓名以便 memory 與 demo 引用，背景人物隨機。

初始密碼

所有 seed 出來的帳號用 Demo!2026 當初始密碼，方便 demo 中切換登入。

注意 Seed 不要動 INITIAL_ADMIN_*（你原本的 admin 帳號）。Seed 用獨立的 demo 用 admin（如 demo_admin），跟真實 admin 隔離。

業績權重模型

產生銷貨單時，每月按權重隨機挑業務員：

weight = base_weight[archetype] * seasonal_multiplier(month) * personal_event(name, month)
# 例：明星 #2 在 2025-08/09 個人事件 = 0.2（請假）
# 例：新人崛起在 2026-04 起 personal_event = 2.5
為什麼這樣設計 Demo 中常見問題：「業務 A 的銷售環比是進步還是退步？」、「哪些業務需要關心？」、「為什麼某月業績下滑？」——這些都需要業務員之間有可解釋的差異。

J · 事件
事件 特殊事件編劇（10+ 個劇本伏筆）
  1. RTX 5090 上市熱（2025-01）：銷量爆衝、毛利率高，但 supply 緊（每月只能拿到 30 張左右），常缺貨。
  2. RTX 4090 清庫存（2025-Q2）：售價從 NT$ 75,000 → 68,000，毛利率被壓縮，週轉變快。
  3. AI 熱潮 H100 漲價（2024-12 ~ 2025-Q3）：盛達科技把 H100 進價拉高 25%，我們也跟著漲，但毛利率反而擴大（賣方市場）。
  4. AI 回穩（2025-Q4 起）：H100 進價回到原價、毛利率 normalize。

L · 技術
技術 技術實作（API + timestamp backfill）

API 寫入流程（每月迴圈）

  1. 登入 POST /api/v1/auth/login 拿 admin token，後續所有寫入都帶這個 token。
  2. 本月進貨：對每張要建的進貨單，POST /purchase-orders 建 draft → POST /purchase-orders/{id}/receive 入庫（觸發 stock 加 + cost_price 回寫 + AP 建立）。
  3. 本月銷貨：對每張要建的銷貨單，POST /sales-orders（指定該月活躍的 salesperson）→ POST /sales-orders/{id}/confirm（觸發 stock 扣 + unit_cost 快照 + AR 建立）。
  4. 本月盤點（如有）：POST /stock-adjustments。
  5. 本月收付款：POST /ar-payments 與 POST /ap-payments（依客戶/供應商付款習慣推進 paid_amount）。
  6. Backdate：本月所有單據建完，跑一個內部 SQL 把該批 records 的 ordered_at / confirmed_at / received_at / paid_at / adjusted_at 推到目標月份內隨機日期。

為何要 backdate

FastAPI 端 SalesOrderCreate 雖然接受 ordered_at，但 confirmed_at 由 confirm() 在 server 端寫死為 datetime.now()。要把整批 confirmed_at 推到歷史，只能 raw SQL UPDATE。AR issued_at / due_date、payment 的 paid_at 同理。

Backdate SQL 範例

# 把某批 SO 的 confirmed_at 推到 2025-03 內
UPDATE sales_orders SET
  confirmed_at = :target_dt,
  ordered_at = :target_dt - interval '2 day',
  created_at = :target_dt - interval '2 day',
  updated_at = :target_dt
WHERE id IN (:ids);

# SQLite 沒有 interval — 用 datetime() 函式或 Python 端算好再傳

API client 設計

# seed/generators/api_client.py
class ApiClient:
    def __init__(self, base_url: str, db_url: str): ...
    def login(self, username: str, password: str) -> None: ...
    def post(self, path: str, json: dict) -> dict: ...
    def get(self, path: str, params: dict = None) -> dict: ...

    # Direct DB access for things API can't do
    def backdate_orders(self, table: str, ids: list, target_dt: datetime): ...
    def raw_sql(self, sql: str, params: dict): ...

效能估計

  • 每 API call ~50-100 ms（local + SQLite）→ 1,200 個 POST ≈ 1-2 分鐘。
  • 2 分鐘 backfill 內可接受、不用做 batch 優化。
  • 如後續變慢可加 concurrency（asyncio + httpx async），但 demo 用初版 sync 即可。
注意 Backend 必須在 seed 跑前先起來 (uvicorn run + alembic upgrade head + 初始 admin 種好)。seed.py 要檢查 /api/v1/health 確認 backend 可達。

def main():
    args = parse_args()
    client = ApiClient(API_BASE_URL, DB_URL)

    if args.reset:
        wipe_demo_data(client)         # 只清 is_demo 或 demo_admin 建的

    client.login("demo_admin", ADMIN_PASSWORD)

    # 一次性建立：角色、員工、供應商、客戶、商品
    seed_setup(client)
    seed_people(client)                # 50 staff
    seed_catalog(client)               # 8 suppliers, 30 customers, 33 products

    # 18 月時序主迴圈
    for ym in months_from("2024-12", "2026-05"):
        if args.resume and already_done(ym):
            continue
        with month_checkpoint(ym):
            generate_purchases(client, ym)
            generate_sales(client, ym)
            generate_adjustments(client, ym)
            generate_payments(client, ym)
            backdate_month(client, ym)
            apply_special_events(client, ym)  # 漲價、缺貨、報廢...

    print_summary()

N · 實作分階段
路線 實作分階段（6 步、預估 ~6 小時）

給 Claude（或下一位開發者）按此順序實作，每一步結束都能獨立驗收。

Step 1 — 骨架（~30 分鐘）易

建好 seed/ 內的所有空殼檔案，讓 python seed.py --dry-run 能執行（即使只 print 「no-op」）。

  • requirements.txt：httpx、python-dotenv、sqlalchemy（少量依賴）
  • .env.example：API_BASE_URL、SEED_DB_URL、ADMIN_USERNAME、ADMIN_PASSWORD
  • seed.py：CLI 入口（argparse 解析 --reset / --resume / --dry-run）
  • generators/api_client.py：ApiClient class（login / post / get / raw_sql / backdate）
  • config/__init__.py、generators/__init__.py 空模組
  • data/ 空目錄（之後放 gpu_prices.json）

驗收：python seed.py --dry-run 印出計劃綱要而不出錯。

Step 2 — 靜態資料（~1 小時）易

把計劃書 C-H 章節的內容寫成 Python data structures。純資料、無邏輯。

  • config/products.py：33 SKU dict（sku / name / category / cost / price / launch_date）
  • config/suppliers.py：8 家供應商（名稱 / 主要產品 / 付款條件 / 劇本旗標）
  • config/customers.py：30 家客戶（11 劇本角色 + 19 背景）
  • config/people.py：50 員工（姓名 / 角色 / 業績檔次 / 個人事件）
  • config/stories.py：時序事件編劇表（漲價、缺貨、客戶流失等）

驗收：寫個 tests/test_config_shape.py 確認 count 對（33 商品、8 供應商、30 客戶、50 員工）+ schema 合法。

Step 3 — 一次性建立（~1 小時）易

透過 API 把靜態資料推進 backend。

  • generators/seed_setup.py：確保 4 角色存在；用 INITIAL_ADMIN 登入後建 demo_admin、登入 demo_admin 後續所有寫入用此 token
  • generators/seed_people.py：透過 POST /users 建 50 人
  • generators/seed_catalog.py：建 5 分類 → 33 商品、8 供應商、30 客戶

驗收：登入 demo_admin 後 GET /users、/products、/suppliers、/customers 都看得到資料。

Step 4 — 時序主迴圈（~2 小時）中

對 18 個月跑迴圈，產出進銷貨單。本步驟最複雜也最有戲。

  1. generators/seed_timeline.py 對每個 (year, month)：
    • 依季度乘數決定本月銷貨單目標數量（base × seasonal × event_modifier）
    • 對每張要建的 SO：依檔次權重挑業務員、依客戶活躍度挑客戶、依商品熱度挑 product mix、配 unit_price（含議價兇折扣）
    • 對應產生 PO（先進貨後銷貨，確保庫存足夠 confirm）
    • POST /purchase-orders → .../receive → POST /sales-orders → .../confirm
  2. generators/timestamp_backfill.py：本月跑完，用 raw SQL UPDATE 把該批 records 的 timestamp 推到目標月份內隨機日期
  3. 寫 checkpoint 檔（.checkpoint）

驗收：跑完 1 個月後，GET /inventory/monthly-report?year=2024&month=12 看到資料；跑完全部 18 月後，月報每月都有資料且銷量呈現 Q4 高峰。

Step 5 — 特殊事件注入（~1 小時）中

讓「劇本」真的反映在資料裡。每個事件對應一段邏輯。

  • 大宇 2026-04/05 漲價：在生成那兩個月的進貨單時，把對應 SKU 的 unit_cost 拉高 10% / 8%（再 receive 會自動回寫 product.cost_price）
  • AI 熱潮：2024-12 ~ 2025-Q3 期間，盛達科技進的 H100/A100 cost_price 高 25%、2025-Q4 回正
  • 明星 #2 請假（2025-08, 09）：那兩個月李曉雯的 weight 設 0.2
  • 新人崛起：蔡明軒在 2026-01-03 weight = 0.3、2026-04-05 weight = 2.5
  • 祥豐流失：2026-03 後祥豐客戶活躍度設 0.1
  • 缺貨事件：2025-09 / 2026-02 跳過 NV-5070、NV-5070Ti 的進貨單，造成自然缺貨
  • generators/seed_adjustments.py：在散落月份注入 6+ 筆盤點（盤盈 / 盤虧 / 報廢各至少 1）
  • generators/seed_payments.py：依客戶付款習慣產生收款記錄（旭光、祥豐故意拖到逾期），含 2 筆作廢演示

驗收：跑 GET /analytics/margin/trend?months=18 看到 2026-04/05 毛利率明顯下滑；GET /accounts-receivable/aging 看到 60+ 天 bucket 有未結清 AR。

Step 6 — 收尾 + 文件（~30 分鐘）易

  • Checkpoint 機制完整實作（--resume 可從中斷處繼續、--reset 全砍重來但保留 INITIAL_ADMIN）
  • seed/README.md：給使用者的快速啟動說明（含環境變數、執行步驟、debug tips）
  • 對照計劃書 section N（驗收條件）逐項勾選；任一項不過關回去調 stories.py / weights
  • 記憶體更新：新增 project-seed-data.md 記錄 seed 設計決策與 demo 角色名單

驗收：10 條驗收條件全綠。

並行 / 切割建議

  • 單 session 完成：Step 1+2+3（~2.5 小時），讓使用者第一次能看到 50 人 + 33 商品 + 30 客戶在系統中。
  • Step 4 一個 session：時序主迴圈是核心、需要不被打斷。
  • Step 5+6 一個 session：把劇本注入 + 驗收一起做。
debug 小提醒 Step 4 最容易卡：若 confirm 失敗（庫存不足），表示該月進貨單先後順序或數量設計錯。建議實作時加 --month=2024-12 旗標只跑單月，方便迭代調整。

P · 不做
不做 不做的東西
  • 不做 Faker 大量灌水——產量靠劇本控制，不用 1 萬個假客戶。
  • 不做完整 i18n 客戶名稱——seed 只用中文姓名，demo 受眾就是中文。
  • 不做多倉——backend 還沒支援、demo 用不到。
  • 不做退貨單獨流程——用 stock_adjustment with reason=scrap/shortage 模擬。
  • 不做匯率欄位——backend 沒這欄位、價格直接寫成已換算後的 NTD。
  • 不做員工照片 / avatar——只給 username + full_name。
  • 不做訂單變更歷史——backend 沒做、且本 demo 不講審計。
  • 不做 GPU 廠商型號（ASUS ROG / MSI / Gigabyte）——SKU 用 NVIDIA / AMD 公版命名，簡化分類。
